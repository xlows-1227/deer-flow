# 每日按人记忆总结产品需求

## 背景

OpenAI 2026 年的记忆更新强调后台异步合成、记忆新鲜度、用户可见的记忆摘要，以及可解释的记忆来源。DeerFlow 目前已经具备按用户存储记忆、异步更新记忆、可编辑事实、Prompt 注入等基础能力，但当前流程会把会话中提取到的信息直接写入长期记忆画像。

建议以“每日按人总结”为核心重建 DeerFlow 记忆系统，并逐步替换当前直接写入 `memory.json` 的旧记忆机制。每日总结用于保存当天有价值的上下文和证据，再由新的合并流程生成可注入 Prompt 的长期记忆视图，避免把临时计划、过期位置、一次性上传文件事件或噪声对话直接污染长期记忆。

## 目标

构建一套新的记忆系统：先把每个用户一天内的交互总结为可审阅的每日记忆，再周期性地把稳定模式和高置信事实合并为新的长期记忆视图，并最终替换当前 `MemoryUpdater` 直接维护 `memory.json` 的实现。

核心目标：

- 按 `user_id + local_date` 生成用户级记忆。
- 用新的“每日总结 + 长期视图 + 来源索引”作为主存储和主注入来源，替换现有 `user/history/facts` 直接滚动更新模型。

辅助目标：

- 通过临时上下文的自然过期，提高记忆新鲜度。
- 保留长期事实到每日总结、线程和运行记录的来源追溯。
- 通过审阅、纠正、删除，以及未来的“不再提及”能力，让用户保持控制权。
- 为旧记忆数据提供迁移路径，避免升级后丢失已有用户画像和 facts。

## 非目标

- 不把完整向量检索作为第一版的主记忆机制。
- 不把所有每日总结都注入到每次 Prompt 中。
- 不持久化临时上传路径或仅在单次会话有效的文件事件。
- 不存储敏感信息，除非它明确有用、用户认可，并且可以通过删除控制管理。
- 不要求第一版一次性删除所有旧代码路径；可以先通过兼容适配层平滑切换。

## 当前系统

当前 DeerFlow 记忆系统已有以下基础能力：

- `MemoryMiddleware` 在 agent 运行完成后捕获会话。
- `MemoryUpdateQueue` 通过 debounce 合并更新。
- `MemoryUpdater` 使用 LLM prompt 更新用户摘要、历史摘要和 facts。
- `FileMemoryStorage` 存储按用户隔离的记忆 JSON；旧系统中曾支持 agent 级记忆，但新系统第一版不继续做 agent 级隔离。
- 前端 Memory 设置页支持审阅、导入导出、清空和 fact 增删改。
- Prompt 注入逻辑会把当前画像和高置信 facts 格式化进 `<memory>`。

主要缺口是：会话级更新会直接压缩进长期画像。这会让新鲜度管理、来源审阅和时间衰减变得困难，也使得未来很难解释“某条记忆来自哪里、为什么还在生效”。

## 方案模型

替换后的新记忆模型：

```text
会话消息
  -> 每日按人总结
  -> 长期记忆视图
  -> 来源索引与用户控制
  -> 按相关性注入 Prompt
```

每日总结是证据层；长期记忆视图是面向 Prompt 的紧凑个性化层；来源索引用于审阅、纠错、删除和解释。旧 `memory.json` 可以在迁移期作为导入来源或兼容输出，但不再作为新系统的主写入目标。

## 每日总结结构

第一版建议结构：

```json
{
  "version": "1.0",
  "id": "daily_2026-06-05_default",
  "personId": "default",
  "date": "2026-06-05",
  "timezone": "Asia/Shanghai",
  "summary": "",
  "workContext": [],
  "preferences": [],
  "durableFacts": [],
  "temporaryContext": [],
  "openLoops": [],
  "corrections": [],
  "sourceThreads": [],
  "sourceRuns": [],
  "updatedAt": ""
}
```

字段含义：

- `summary`：当天真正重要内容的简短叙述。
- `workContext`：当天出现的项目、仓库、工具、约束和决策。
- `preferences`：用户明确表达或强烈暗示的偏好。
- `durableFacts`：长期记忆候选事实。
- `temporaryContext`：计划、截止时间、位置、短期任务和进行中的事项。
- `openLoops`：未完成请求或后续待办。
- `corrections`：用户纠正或需要避免重复的 agent 错误。
- `sourceThreads/sourceRuns`：用于审阅和调试的来源追溯。

字段内容必须遵守安全边界：每日总结应偏向用户画像、兴趣爱好、偏好、关注点、近期活动概览和 skill 使用习惯，不记录具体任务结果、具体问题细节、连接信息或可暴露环境的元数据。

## 存储

推荐保持本地优先的文件布局：

```text
users/{user_id}/memory/profile.json
users/{user_id}/memory/daily/{YYYY-MM-DD}.json
users/{user_id}/memory/sources.jsonl
users/{user_id}/memory/legacy-memory.json
```

这样可以复用现有按用户隔离的模型，同时把新系统的主存储从旧 `memory.json` 中拆出来。`legacy-memory.json` 仅用于迁移备份或回滚，不参与新写入链路。

## 处理流程

1. 在一次 run 完成后捕获会话事件。
2. 按 `user_id`、`thread_id` 和日期入队；当前实现以 UTC 日期为稳定默认值，后续再切换为用户本地日期。
3. 每日定时任务将当天会话汇总为 `DailyPersonSummary`。
4. 每个会话/线程支持用户手动触发 rollup，用于将指定会话增量合并到当天记忆；当前同一天同一线程重复触发采用幂等返回，不支持自动重新生成该线程贡献。
5. 运行合并流程，仅当信息具备长期性、重复出现、被明确纠正或明显有用时，才更新新的长期记忆视图。
6. 构造 Prompt 时，注入长期画像，并在确实有帮助时附加最近且相关的每日总结片段。
7. 迁移期由兼容层把新长期记忆视图转换为旧 API 需要的响应结构；切换完成后再移除旧 `MemoryUpdater` 写入路径。

## 合并规则

满足以下条件时，提升到长期记忆：

- 用户明确陈述稳定事实或偏好。
- 同一偏好或上下文跨多天重复出现。
- 用户纠正了 agent，且该纠正可以避免未来重复犯错。
- 某个项目、工具链或工作流约束在单次会话之外仍保持活跃。

仅保留在每日记忆中：

- 信息是短期任务、当天计划、临时位置或进行中备注。
- 事实对来源审阅有用，但不值得长期注入。
- 置信度较低，或证据只出现过一次。

满足以下条件时过期或改写：

- 时间推进后，未来计划变成历史上下文。
- 临时位置或事件不再适用。
- 新的用户信息与旧记忆冲突。

## 用户控制

第一版：

- 按日期查看每日总结。
- 删除某一天的每日总结。
- 刷新或重新生成某一天的每日总结。
- 保留现有 fact 创建、编辑、删除能力。

后续版本：

- 纠正每日总结中的选中文本。
- 将某些细节标记为“不再提及”。
- 在个性化回答下展示记忆来源。
- 标记某个来源是否相关。

## API 范围

建议新增：

- `GET /api/memory/daily?date=YYYY-MM-DD`
- `GET /api/memory/daily?limit=30`
- `POST /api/memory/daily/rollup`
- `DELETE /api/memory/daily/{date}`
- `POST /api/memory/consolidate`
- `GET /api/memory/profile`
- `POST /api/memory/migrate-legacy`

这些接口应使用 `get_effective_user_id()`，并遵循现有 memory router 的按用户隔离行为。

## Prompt 注入策略

默认 Prompt 注入保持紧凑：

- 始终在配置的 token 预算内包含 `profile.json` 中的长期记忆视图。
- 只有当近期每日上下文与当前请求相关时才注入。
- 优先使用最近 1-7 天的片段，而不是更老的每日总结。
- 永远不注入仅上传文件事件或已经过期的临时上下文。

这样可以让记忆持续有帮助，而不会把 Prompt 变成无边界的历史归档。

迁移完成后的默认注入来源应改为：

- `profile.json` 中的长期记忆视图。
- 与当前请求相关的近期每日总结片段。
- 高优先级 corrections 和“不再提及”规则。
- 必要时附带来源引用，但不把来源全文默认注入。

## MVP 实施计划

第一阶段：

- 新增每日总结存储 helper。
- 新增每日 rollup prompt 和 updater。
- 新增读取、rollup、删除 API。
- 新增 `profile.json` 长期记忆视图结构。
- 新增从旧 `memory.json` 到新结构的迁移脚本或 API。
- 增加按用户隔离、日期隔离、上传过滤和 JSON 持久化测试。

第二阶段：

- 从每日总结合并长期信息到新的 `profile.json`。
- 为 facts 增加指向每日总结和线程的来源元数据。
- 在前端 Settings > Memory 中增加每日总结审阅区域。
- 将 Prompt 注入切换为读取新长期记忆视图。

第三阶段：

- 增加按相关性选择每日总结并注入 Prompt 的能力。
- 增加个性化回答的来源展示。
- 增加纠正和“不再提及”控制。
- 下线旧 `MemoryUpdater` 直接更新 `memory.json` 的写入链路，仅保留必要的兼容读取或迁移能力。

## 成功指标

- 用户在新线程中不需要重复稳定偏好。
- 临时计划过期后不再影响回答。
- 记忆事实可以追溯到来源日期、线程或 run。
- 每日总结对用户来说可审阅、可理解。
- Prompt 中的记忆内容保持在配置的 token 预算内。
- 新系统上线后，旧 `memory.json` 不再是主写入目标。

## 已确认决策

- 新记忆系统的目标不是增强现有 `memory.json` 流程，而是替换当前记忆系统。
- 新系统以“每日按人总结”为核心证据层。
- rollup 触发方式：每日定时触发；同时每个会话/线程支持用户手动触发。
- 前端展示范围：用户可以同时看到“每日总结 + 长期画像”。
- 删除语义：删除每日总结默认采用软删除；该来源立即退出展示、注入和后续合并，并异步重算长期画像。若长期记忆仍有其它来源支撑则保留，仅移除被删来源引用；若只由被删来源支撑则移除。彻底删除作为二级操作；“不再提及”使用 suppression/tombstone 规则，不等同于删除来源。
- 隔离模型：第一版只做用户级记忆，不做 agent 级记忆隔离；通用 agent 的具体使用场景由当前会话显式指令控制。
- 后续文档默认使用中文保存。

## 待确认问题

### 产品与体验

- “不再提及”是只影响 Prompt 注入，还是同时从来源和长期画像中删除？
- 用户手动编辑 fact 或 summary 时，是否需要写入一条合成来源记录，方便之后审计？

### 数据与迁移

- 旧 `memory.json` 迁移策略：自动迁移、用户确认后迁移，还是首次启动时备份并迁移？
- 旧 facts 中缺少来源证据的内容，迁移后应标记为 `legacy`，还是要求重新从历史线程中生成来源？
- 每日总结默认保留多久：永久、本地保留 N 天，还是可配置？
- `profile.json` 是否需要保留与旧 `user/history/facts` 兼容的字段，方便 API 和前端平滑迁移？

### 注入与检索

- 第一版相关性选择用规则和关键词即可，还是需要直接引入 embedding/vector index？
- 每次 Prompt 中每日总结片段的 token 预算上限是多少？
- corrections 是否应拥有最高注入优先级？
- “当前活跃项目/未完成事项”是否要单独作为 top-of-mind 视图维护？

### 安全与合规

已确认边界：

- 优先总结用户的兴趣爱好、偏好和用户画像。
- 可以总结用户关注的主题，以及最近大致在做什么。
- 可以总结用户使用 skill、工具、文档、代码流程等方面的习惯和偏好。
- 不总结具体事情的结果、具体问题细节、失败原因、排查过程或业务数据内容。
- 不记录连接信息、账号、路径、URL、密钥、token、数据库信息、文件元数据、运行环境元数据等可能暴露系统或数据访问面的内容。
- 如果某条信息既有长期偏好价值又包含敏感细节，应只保留抽象偏好，丢弃细节。例如保留“用户经常分析数据库连接类问题”，不保留具体 host、库名、表名、连接串或报错。

## 2026-06-06 产品状态更新

### 已落地范围

- 新记忆系统已切换为以“每日总结 + 长期画像”为核心的用户级记忆模型。
- 每日总结支持后台定时汇总、设置页立即汇总，以及历史会话列表中的单会话汇总。
- 设置页仅展示与新记忆模型相关的长期画像、每日总结和手动记忆，旧上下文与历史分区不再作为主要界面展示。
- “清空全部记忆”语义已明确为清除当前用户的全部记忆相关内容，包括长期画像、每日总结、rollup 输入、来源记录、手动记忆和旧版记忆文件。
- 删除每日总结后会立即退出展示和长期画像合并，并重新生成长期画像。
- 长期画像重建时会保留用户手动添加的记忆。
- 历史会话支持“总结到记忆”，仅总结所选会话，并增量合并到当天总结，不覆盖当天其它会话形成的记忆。
- 单会话重复汇总采用幂等语义：同一会话已进入当天总结后，重复点击不会重复添加相同内容。

### 最新记忆证据规则

记忆汇总只允许使用用户真实表达作为证据：

- 仅用户消息可以成为自动记忆证据，助手回复不作为用户画像或偏好的证据。
- 必须移除会话中注入的 `<system-reminder>`、`<memory>` 等旧记忆上下文，禁止记忆系统总结并强化自己。
- 问候、确认、无有效画像信息的对话不应产生每日总结。
- 每条记忆只表达一个维度，并只进入一个分类，避免同一含义同时出现在偏好和使用习惯中。
- 使用习惯和输出偏好应分开表达。例如：
  - 使用习惯：`使用 AI 总结业务文档`
  - 输出偏好：`文档总结结果偏好 Markdown 格式`
- 不应生成混合表达，例如：`利用 AI 对业务文档进行总结并输出为 Markdown`。该表达同时包含使用习惯和输出偏好，容易与其它事实重复。

### 立即汇总行为

- 设置页“立即汇总”会先刷新当前用户尚未落盘的记忆捕获队列，再汇总当天全部可用输入。
- 历史会话“总结到记忆”直接读取所选会话的最新 checkpoint，仅汇总该会话内容。
- 无有效用户证据时返回空结果，并提示“暂无可汇总的会话内容”。

### 当前产品限制

- 已进入每日总结的历史重复内容不会被自动语义合并；用户可以删除对应每日总结后重新汇总。
- 当前幂等策略以 `date + thread_id` 为判断依据。已汇总会话后续新增消息时，不会自动重新合并该会话，需要后续设计“强制重新汇总”或版本化来源。
- “不再提及”、来源查看、每日总结内容编辑仍未进入当前已落地范围。
