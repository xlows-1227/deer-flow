# 文档语言约定

## 文档优先原则

本项目所有产品、设计、代码、配置和行为更新，都必须优先更新对应文档，再进行实际修改。

执行顺序：

1. 先在对应分类文档中记录需求、决策、方案或变更范围。
2. 再进行代码、配置、界面或其它实现修改。
3. 实现完成后，更新执行记录、验证结果、已知限制和后续事项。
4. 如果实现与原文档方案不一致，必须同步修正文档，禁止保留互相冲突的说明。

文档分类：

- 产品目标、用户体验、范围和验收口径优先更新到 `docs/product`。
- 架构、接口、数据模型和技术决策优先更新到 `docs/design`。
- 实施步骤、代码变更、测试结果和问题记录更新到 `docs/execution`。

## 语言约定

本项目后续新增或更新的内部文档默认使用中文，包括产品需求、设计方案、开发计划、执行记录、测试说明、变更总结和运维说明。

修改既有英文内部文档时，应在不影响上游同步和对外契约的前提下逐步改为中文。文档评审时，正文存在无必要的英文段落视为需要修正的问题。

## 例外

以下内容可以保留英文：

- API 路径、字段名、配置项、代码符号和文件路径。
- 命令、代码块、错误码、提交信息和测试名称。
- 引用的外部标题、协议名、专有名词。
- 需要与上游英文规范保持一致的短语。
- 面向国际用户或需要与上游仓库保持同步的公开文档；此类文档应同时提供或维护对应中文版。

## 目录分类

- `docs/product`：产品需求、范围说明、用户价值和验收口径。
- `docs/design`：架构设计、技术方案、规格说明。
- `docs/execution`：实施计划、执行记录、验证证据和变更总结。
